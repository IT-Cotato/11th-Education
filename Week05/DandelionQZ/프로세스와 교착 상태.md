# 프로세스

### 프로그램 vs 프로세스

프로그램은 디스크 상의 실행 가능한 파일로 컴퓨터에 저장된 코드.

프로세스는 프로그램이 메모리에 로드되어 실행 중인 상태. 

프로세스는 운영체제에 의해 관리되며, 메모리에 올려진 데이터이면서 독립적으로 실행되고 자원을 할당 받을 수 있는 단위.

<br></br>

프로그램은 실행 파일, 소스 코드, 바이너리 코드 등과 같은 속성을 가짐.

프로세스는 프로세스 ID, 프로세스 상태, 메모리 주소, 스택, 힙 등의 속성을 가짐.

프로세스는 시스템에서 실행 중인 프로그램의 고유한 인스턴스이며, 각 프로세스는 고유한 가상 메모리 공간을 가짐. 

- stack : 함수가 실행되는 동안 지역변수가 저장되는 임시 공간 (지역변수, 매개변수, 리턴값 등)
- heap : 프로그램 실행동안 동적으로 할당되는 변수가 저장되는 공간
- data : 전역 변수를 저장하는 공간. 프로그램 시작시 초기화되며 프로세스가 종료될 때까지 유지됨.
- text(code) : 기계어로 컴파일된 코드가 저장되는 공간

<br></br>

### PCB와 문맥 교환

프로그램이 실행되면 운영체제는 프로세스의 정보를 저장하는 PCB(Process Control Block)라는 별도의 공간을 생성함. 그리고 프로세스가 종료되면 해당 PCB를 해제함.

PCB는 운영체제 커널 영역에 저장됨. 즉, 프로세스의 가상 메모리 영역에 존재하지 않음. 

PCB는 프로세스에 대한 모든 정보가 모여있는 곳으로, TCB(Task Control Block)라고도 함. 

<br></br>

PCB에 저장되는 정보

- PID
    
    PID(Process ID)는 운영체제에서 프로세스를 식별하기 위해 프로세스에 부여하는 번호.
    
- 프로세스 상태
    
    프로세스의 현재 상태(생성, 준비, 실행, 대기, 종료 중 하나).
    
- CPU 스케줄링 정보
    
    프로세스 우선순위, 할당된 cpu 시간 등 스케줄링 관련 정보. 
    
- CPU 레지스터
    
    프로세스의 레지스터 상태. CPU 내 범용 레지스터 (AX, BX, CX, DX), 데이터 레지스터(SP, BP, SI, DI), 세그먼트 레지스터(CS, DS, ES, SS) 등이 갖고 있는 값. 
    
    - PC (Program Counter)
        
        다음에 실행 할 명령어의 주소를 가리키는 레지스터.
        
        코드를 실행시키면 코드가 한 줄씩 내려가며 실행되는데, 프로세스는 한 줄을 처리 후 다음 실행할 코드가 어딘지 알려줘야 코드를 찾아서 실행할 수 있다. 이때, 다음 실행할 코드의 주소를 저장하는 레지스터가 PC이다.
        
        PC는 프로그램의 흐름을 제어하는 역할을 한다. 프로세스가 중단되거나 인터럽트(예외 상황)가 발생했을 때 현재 실행 중이던 명령어의 주소를 저장하고, 다시 프로세스가 실행될 때 주소를 찾아가 재실행 시킬 수 있는 역할을 한다.
        
    - SP (Stack Pointer)
        
        현재 실행 중인 프로세스의 스택(Stack)의 최상단을 가리키는 레지스터. 
        
        코드를 작성할 때, 재귀함수처럼 함수 안에 함수를 호출해야하는 경우가 있다. 이럴 경우, 가장 최근에 호출한 함수 순서대로 Stack에 쌓이게 되는데, 가장 최근에 호출된 함수(stack의 최상단 함수)의 위치를 알고 있으면, 어디까지 함수가 호출되어 있는지 프로세스가 알 수 있다.
        
        SP는 프로세스의 스택 영역(가상 메모리의 stack 영역)을 관리하고, 스택 오버플로우(Stack Overflow, 스택 영역 밖으로 메모리가 넘어가는 현상)와 같은 문제를 방지하기 위해 제한된 메모리 영역을 사용하는 등의 역할을 한다. 
        
- 입출력 상태
    
    프로세스가 현재 사용 중인 입출력 장치, 프로세스 수행 시 필요한 주변 장치와 파일의 정보. 어떤 입출력 요청을 보내고 있는지, 어떤 파일을 열어두었는지 등.
    
- etc

<br></br>

‘작업 관리자’ 창에서 현재 실행 중인 프로세스들을 모두 확인 가능. 

PID, 프로세스들의 상태, 계정 정보, 사용 중인 메모리 할당 정보 등등 필요한 정보를 확인할 수 있는데, 이 정보들을 PCB에서 불러옴. 

<br></br>

프로세스 상태 생명주기

1. 생성(new)
    
    프로세스가 생성된 상태. 아직 실행되기 위한 자원은 할당받지 못함. 
    
2. 준비 (Ready)
    
    프로세스가 실행을 기다리는 상태. 프로세스가 필요한 자원을 모두 할당받아 실행을 위한 준비가 완료됨. 하지만 CPU는 할당받지 못함. 이 상태에서는 CPU를 할당받기 위해 스케줄링 대기열(Queue)에 들어감. 
    
3. 실행 (Running) 
    
    프로세스가 CPU를 할당받아 실제로 코드를 실행하는 상태. 프로세스는 작업을 처리하고 결과를 생성. 
    
4. 대기 (Blocked) 
    
    프로세스를 처리중에 작업 시간이 초과되거나 자원 사용을 위해 대기해야 하는 이벤트가 발생해서 프로세스가 잠시 멈춘 상태. 이 상태에서는 CPU를 사용하지 않음. 특정 자원을 사용할 수 있을 때까지 실행을 멈추고, 조건이 만족돠면 다시 준비 상태로 들어감. 
    
5. 종료 (Terminated, exit)
    
    프로세스의 실행이 완료되어 종료된 상태. 이 상태에서는 할당된 자원이 해제되고, 프로세스의 메모리 공간은 운영체제에 반환됨. 

<br></br>
    

문맥 교환 (Context Switching)

여러 프로세스를 번갈아 실행하기 위해, 현재 실행 중이던 프로세스의 상태를 PCB에 저장하고, 다음 프로세스의 상태를 불러오는 것. 

한 프로세스에 할당된 CPU 사용 시간이 끝나거나, 인터럽트가 발생하거나, 입출력을 위해 대기해야 하는 경우. 

발생 시점 

- 준비 상태 → 실행 상태
- 실행 상태→ 준비 상태
- 실행 상태 → 대기 상태

<br></br>

문맥 교환 시나리오 

1. P0 프로세스가 인터럽트 되면서 PCB0에 P0 프로세스의 상태 정보를 저장한다.

2. 다음 수행할 P1 프로세스의 PCB1에서 P1 프로세스의 상태 정보가 CPU에 재로딩된다.

3. P1 프로세스를 일정 시간 수행한다.

4. P1 프로세스가 인터럽트 되면서, PCB1에 P1 프로세스의 상태 정보를 저장한다.

5. 다음 수행할 P0 프로세스의 PCB0에서 P0 프로세스의 상태 정보가 CPU에 재로딩된다.

6. P0 프로세스를 일정 시간 수행한다.

<br></br>

# 교착상태

교착 상태(Deadlock)는 여러 프로세스가 자원을 할당 받은 상태로 서로가 가진 자원을 기다리면서 영원히 끝나지 않는 상태에 빠지는 것.

<br></br>  

### 교착 상태의 필요조건 4가지

‘Coffman의 4가지 조건’이라고 하기도 함. 

네 가지 조건이 모두 만족되어야 교착 상태가 발생. 즉, 하나라도 해당되지 않으면 교착 상태는 발생하지 않음. 

- 상호 배제(Mutual Exclusion)
    
    자원은 한 번에 하나의 프로세스만 사용할 수 있다 
    
- 점유와 대기(Hold and Wait)
    
    자원을 가진 상태에서 다른 자원을 기다린다
    
- 비선점 (No Preemption)
    
    자원을 강제로 빼앗을 수 없다
    
- 원형 대기(Circular Wait)
    
    자원이 서로 순환하며 기다리는 상황

<br></br>
    

### 교착 상태 해결방법 4가지

- 예방 (Prevention)
    
    네 가지 조건 중 하나를 만족하지 않도록 설계하기. 
    
    실효성이 적어 잘 사용되지 않음. 
    
    - 상호 배제 예방
        
        시스템 내의 상호 배타적인 모든 자원을 공유하도록 함. 작업 결과에 영향을 미칠 수 있어 어려움. 
        
    - 점유와 대기 예방
        
        전부 할당하거나 아예 할당하지 않음. 사용하려는 모든 자원을 알기 어렵고, 자원의 활용성이 떨어지고, 많은 자원을 사용하는 프로세스가 적은 자원을 사용하는 프로세스보다 불리하고, 결국 일괄 작업 방식으로 동작하게 되어 어려움. 
        
    - 비선점 예방
        
        모든 자원을 빼앗을 수 있도록 함. 임계구역을 보호할 수 없어 어려움. 
        
    - 원형 대기 예방
        
        자원에 숫자를 부여하고 프로세스들은 한 쪽 방향으로 자원을 이용하도록 함. 프로세스 작업 진행에 유연성이 떨어지고, 자원의 번호 부여 방식이 중요해 어려움. 
        
- 회피 (Avoidance)
    
    교착 상태가 발생할 수 있는 상황을 미리 예측해서 피하는 방법. 
    
    자원을 얼마만큼 할당해야 교착 상태가 발생하지 않는다는 보장이 없어서 실효성이 적음. 
    
    대표적인 알고리즘 : Banker’s Algorithm
    
    마치 은행이 대출해줄 때 고객이 나중에 돈을 갚을 수 있는지를 시뮬레이션하듯, 운영체제도 자원을 줄지 말지를 계산하고, 시스템이 안전하다고 판단될 때만 자원을 할당. 
    
    전체 자원의 수와 할당된 자원의 수를 기준으로 안정상태(Safe state)와 불안정 상태(Unsafe state)를 나눔. 불안정 상태 ≠ 교착 상태. 
    
    자원을 요청하는 프로세스의 Expect 값이 Available을 초과할 경우자원을 할당해주지 않음. 반대로 Expect 값이 Available을 초과하지 않는 Safe State에만 자원을 할당해줌. 
    
    모든 프로세스가 데드락을 발생시키지 않으며 정상적으로 실행하고 종료할 수 있는 순서를 Safe Sequence라고 함. 
    
- 검출과 회복 (Detection and Recovery)
    
    교착 상태가 발생해도 괜찮다고 보고, 정기적으로 검사해서 복구하는 방법.
    데드락이 검출되면 자원을 회수하거나, 프로세스 중 일부를 강제로 종료시켜서 해결.
    
    현실적인 접근 방법. 
    
    - 검출 방법 1 - 타임아웃
        
        일정 시간 동안 작업이 진행되지 않은 프로세스를 kill. 교착 상태가 자주 발생하지 않을 것이라는 가정하에 사용. 엉뚱한 프로세스가 강제 종료될 수 있고, 모든 시스템에 적용할 수 없음. 따라서  DB에서는 check point와 roll back을 이용. 
        
    - 검출 방법 2 - 자원 할당 그래프
        
        사이클이 존재하지 않으면 교착상태 아님. 사이클이 존재하면 교착상태 O (사이클이 있지만 교착 상태가 아닌 경우도 있음). 
        
    - 회복 방법
        
        교착 상태를 일으킨 모든 프로세스를 동시에 종료하거나 다음 기준에 따라 프로세스 중 하나를 종료함.
        
        1. 우선순위가 낮은 프로세스를 먼저 종료
        2. 우선순위가 같으면 작업 시간이 짧은 프로세스를 먼저 종료
        3. 위의 두 조건이 같으면 자원을 많이 사용하는 프로세스를 먼저 종료 
- 무시 (Ignore)
    
    대부분 교착 상태가 잘 발생하지 않기에 가장 흔한 방법. 교착 상태의 발생 확률이 비교적 낮으면 별 다른 조치를 취하지 않음. 실제로는 리눅스나 윈도우 등 대부분의 운영체제에서 이런 전략을 사용함.
