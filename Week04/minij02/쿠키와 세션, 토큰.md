## 1. 문제 상황

- 인증 설계에 따라 사용자 경험 차이 발생
- 로그인 유지 실패, 자동 로그아웃 등 문제는 인증 수단에 따라 달라짐
- 쿠키, 세션, 토큰 설계가 핵심

---

## 2. 쿠키 (Cookie)

### ✅ 개념
- 서버가 브라우저에 저장하는 key=value 형식 데이터
- 모든 요청 시 자동 포함 (`Cookie` 헤더)

### ✅ 활용
1. **세션 관리** – 로그인, 장바구니 등
2. **개인화** – 테마, 언어 등 사용자 설정
3. **트래킹** – 분석 및 광고

### ✅ 인증 흐름
1. 클라이언트가 서버에 요청
2. 서버가 `Set-Cookie`로 응답
3. 브라우저가 쿠키 저장
4. 이후 요청 시 `Cookie` 헤더에 쿠키 포함

### ⚠️ 단점
- 쿠키 탈취 위험 → **보안에 취약**
- 용량 제한
- 브라우저 간 호환성 문제
- 쿠키가 커질수록 **네트워크 부하 증가**

---

## 3. 세션 (Session)

### ✅ 개념
- 서버가 사용자 상태를 저장
- 클라이언트는 세션 ID만 쿠키에 저장
- 서버는 세션 ID로 사용자 정보를 조회

### ✅ 인증 흐름
1. 로그인 → 서버가 세션 ID 생성
2. 서버에 사용자 정보 저장
3. 클라이언트는 세션 ID를 쿠키에 저장
4. 요청 시 세션 ID 전송 → 서버에서 대조 후 인증

### ⚠️ 단점
- 세션 ID 탈취 시 사용자 위장 가능
- 서버에 저장소 필요 → **부하 증가**
- 사용자 수 증가 시 확장성 문제

---

## 4. 토큰 (Token)

### ✅ 개념
- Stateless 방식 → 서버는 상태 저장 안 함
- 클라이언트가 토큰을 저장하고 직접 제출
- 서버는 토큰 검증만 수행

### ✅ 인증 흐름
1. 로그인 성공 시 토큰 발급
2. 클라이언트가 토큰 저장
3. 요청 시 `Authorization: Bearer <token>` 전송
4. 서버는 토큰 검증 후 인증

### ⚠️ 단점
- 토큰 길이 증가 → **네트워크 부하**
- Payload는 암호화 X → 민감 정보 저장 금지
- 탈취 시 강제 만료 어려움

---

## 5. JWT (JSON Web Token)

### ✅ 개념
- JSON 기반의 구조화된 토큰
- Header, Payload, Signature 구조
- 전자서명으로 위변조 방지

### ✅ 구조
- **Header**: alg, typ
- **Payload**: 사용자 정보, 권한, 만료 등
- **Signature**: 서버 비밀 키로 서명

### ✅ 장점
- Stateless 인증 → 세션 저장소 불필요
- 빠른 인증 (DB 조회 없음)
- 확장성 우수 (SNS 로그인 등)
- 위변조 방지 가능

### ⚠️ 단점
- 토큰 크기 큼 → **트래픽 부하**
- Payload는 디코딩 가능 → 민감 정보 저장 금지
- 강제 만료 어려움 → 탈취 시 위험

---

## 6. JWT 보완 전략: Access Token + Refresh Token

### ✅ Access Token
- 짧은 유효시간 (예: 15분)
- API 요청 시 사용

### ✅ Refresh Token
- 긴 유효시간 (예: 7일~30일)
- Access Token 재발급에 사용
- 안전한 저장 필요

### ✅ 인증 흐름
1. 로그인 시 두 토큰 발급
2. Access Token으로 인증 수행
3. 만료 시 Refresh Token으로 재발급 요청
4. 서버가 Refresh Token 검증 후 새 Access Token 발급
